# PosterLink E2E 테스트 목록 - 2026-05-09 기준

## 목적

최근 작업된 기능을 기준으로 실제 사용자, 운영자, 관리자, 크롤러 배치 흐름이 깨지지 않는지 확인하기 위한 E2E 테스트 목록이다.

확인 기준은 다음 작업 범위다.

- 포스터 등록 요청 및 포인트 지급
- 온보딩 및 맞춤 설정
- 메인/목록/상세 포스터 노출
- 조회수, 찜 수, 공식 링크 클릭 수 표시
- 마감 제외/마감 포함 토글
- 개인화 필터와 정렬
- 관리자 포스터 검수, 대량 승인/반려, 반려 삭제
- 크롤러 수집, 이미지 필터링, 중복 처리, 자동 분류
- SEO 파일과 배포 기본 동작

---

## 현재 E2E 테스트 상태

이미 Playwright 기반 테스트 구조가 있다.

- 설정 파일: `apps/web/playwright.config.ts`
- 실행 명령: `pnpm --filter web test:e2e`
- 대상 서버: `http://localhost:4000`
- 기존 테스트:
  - `apps/web/e2e/home.spec.ts`
  - `apps/web/e2e/poster.spec.ts`
  - `apps/web/e2e/auth.spec.ts`
  - `apps/web/e2e/ops-access.spec.ts`
  - `apps/web/e2e/ocr.spec.ts`

주의 사항:

- 일부 기존 테스트 파일은 한글 문자열이 깨져 있어 selector 안정성이 낮다.
- `DEADLINE` 같은 예전 문구를 찾는 테스트가 있는데, 현재 UI는 `마감임박`으로 변경되었다.
- 최근 추가된 관리자 검수/크롤러/개인화 정렬 기능은 별도 테스트가 필요하다.

## 테스트 데이터 전제

E2E를 안정적으로 돌리려면 테스트용 seed 또는 fixture가 필요하다.

권장 테스트 데이터:

- 일반 사용자 1명
  - onboarding 완료
  - 기본 지역 1개
  - 관심 카테고리 2개
- 운영자 1명
- 관리자 1명
- 포스터 최소 8건
  - 진행 중 published 3건
  - 마감 published 2건
  - review 2건
  - rejected 1건
- 포스터별 metric
  - 조회수 차이가 나는 포스터
  - 찜 수 차이가 나는 포스터
  - 공식 링크 클릭 수 차이가 나는 포스터
- 크롤러 업로드용 JSON fixture
  - 정상 포스터 이미지 포함 2건
  - 중복 source_key 1건
  - 비포스터 제목 1건
  - 이미지 없음 1건

## P0 필수 테스트

### 1. 공개 페이지 기본 접근

목적:

- 비로그인 사용자가 주요 공개 페이지에 접근할 수 있는지 확인한다.

테스트:

- `/` 접속 시 200 응답
- `/posters` 접속 시 200 응답
- `/privacy` 접속 시 200 응답
- `/terms` 접속 시 200 응답
- `/robots.txt` 접속 시 `Disallow: /admin/` 포함
- `/sitemap.xml` 접속 시 `urlset` 반환

기대 결과:

- 공개 페이지는 로그인 없이 열린다.
- 관리자/운영자/마이페이지/API 경로는 robots에서 제외된다.

### 2. 인증 보호 라우트

목적:

- 비로그인 사용자가 보호 페이지에 접근할 때 로그인 페이지로 이동하는지 확인한다.

테스트:

- `/admin` 접속 시 `/login?redirectTo=/admin` 이동
- `/admin/posters` 접속 시 로그인 이동
- `/operator` 접속 시 로그인 이동
- `/operator/posters/new` 접속 시 로그인 이동
- `/mypage` 접속 시 로그인 이동
- `/favorites` 접속 시 로그인 이동
- `/notifications` 접속 시 로그인 이동

기대 결과:

- 비로그인 사용자는 보호 화면을 볼 수 없다.
- redirectTo가 유지된다.

### 3. 메인 화면 포스터 카운트

목적:

- 메인 상단의 공고 숫자가 실제 화면에 보이는 포스터 수와 맞는지 확인한다.

테스트:

- `/` 접속
- 기본 상태인 `마감 제외`에서 공고 숫자 확인
- 화면에 표시되는 진행 중 포스터 카드 수와 숫자 비교
- `마감 제외` 토글 off
- 마감 포스터가 포함되어 숫자가 증가하거나 동일한지 확인
- 다시 토글 on
- 마감 포스터가 사라지고 숫자가 다시 줄어드는지 확인

기대 결과:

- 상단 공고 숫자는 현재 화면 필터 결과와 일치한다.
- 마감 포함/제외 상태에 따라 숫자가 바뀐다.

### 4. 포스터 목록 마감 토글

목적:

- `/posters` 목록에서 마감 포스터 포함/제외가 정상 동작하는지 확인한다.

테스트:

- `/posters` 접속
- 기본 상태에서 마감 badge가 있는 포스터가 보이지 않는지 확인
- `마감 제외` 토글 off
- 마감 포스터가 목록에 나타나는지 확인
- `Total N Results` 숫자가 변경되는지 확인

기대 결과:

- 토글 상태와 목록이 일치한다.
- 총 결과 숫자도 같이 바뀐다.

### 5. 포스터 목록 정렬

목적:

- 최신, 마감임박, 인기, 조회, 찜, 클릭 정렬이 정상 동작하는지 확인한다.

테스트:

- `/posters` 접속
- `최신` 클릭 시 created_at 내림차순
- `마감임박` 클릭 시 진행 중 포스터 중 application_end_at 오름차순
- 마감 포함 상태에서 마감임박 클릭 시 이미 마감된 포스터가 상단에 오지 않는지 확인
- `인기` 클릭 시 `조회 + 클릭*2 + 찜*3` 기준 내림차순
- `조회` 클릭 시 조회수 내림차순
- `찜` 클릭 시 찜 수 내림차순
- `클릭` 클릭 시 공식 링크 클릭 수 내림차순

기대 결과:

- 각 정렬 버튼이 선택 상태로 표시된다.
- 첫 번째 카드가 해당 기준에 맞게 바뀐다.

### 6. 포스터 목록 개인화 필터

목적:

- 로그인 사용자가 `내 맞춤` 토글로 본인 조건에 맞는 포스터만 볼 수 있는지 확인한다.

테스트:

- 테스트 사용자로 로그인
- 온보딩 완료 상태 확인
- `/posters` 접속
- `내 맞춤` 토글 on
- 사용자 기본 지역 또는 관심 카테고리와 연결된 포스터만 표시되는지 확인
- active badge `내 맞춤` 표시 확인
- badge의 X 클릭 시 필터 해제 확인

기대 결과:

- 전체 포스터가 숨겨지는 방식이 아니라 사용자가 원할 때만 좁혀진다.
- 포스터의 첫 번째 카테고리/지역뿐 아니라 전체 연결 categoryIds/regionIds 기준으로 필터된다.

### 7. 포스터 카드 metric 표시

목적:

- 목록/메인/상세에서 조회수, 찜 수, 공식 링크 클릭 수가 보이는지 확인한다.

테스트:

- `/` 접속 후 첫 포스터 카드에 `조회`, `찜`, `클릭` 값 표시 확인
- `/posters` 접속 후 카드에 metric 표시 확인
- `/posters/[id]` 접속 후 상세 metric 표시 확인

기대 결과:

- 세 화면에서 동일한 metric이 표시된다.
- 숫자가 없을 때는 0으로 표시된다.

### 8. 포스터 상세 조회수 기록

목적:

- 포스터 상세 페이지 진입 시 view log가 증가하는지 확인한다.

테스트:

- 특정 포스터 상세 페이지 접속 전 조회수 기록
- `/posters/[id]` 접속
- `/api/poster-views` 또는 metric API 결과 확인
- 조회수가 1 이상 증가하는지 확인

기대 결과:

- 상세 진입 후 조회수가 증가한다.
- 새로고침/중복 기준 정책이 있다면 정책대로만 증가한다.

### 9. 공식 링크 클릭 기록

목적:

- 공식 링크 버튼 클릭 시 click log가 증가하는지 확인한다.

테스트:

- `/posters/[id]` 접속
- 공식 링크 클릭 수 기록
- 공식 링크 버튼 클릭
- 새 탭 또는 외부 링크 이동 확인
- metric API에서 click count 증가 확인

기대 결과:

- 공식 링크 클릭 수가 증가한다.
- 링크가 비어 있으면 버튼이 보이지 않거나 비활성화된다.

### 10. 찜하기

목적:

- 로그인 사용자가 포스터를 찜하고 해제할 수 있는지 확인한다.

테스트:

- 로그인 사용자로 `/posters/[id]` 접속
- 찜 버튼 클릭
- 찜 수 증가 확인
- `/favorites`에서 해당 포스터 표시 확인
- 다시 찜 버튼 클릭
- `/favorites`에서 사라지는지 확인

기대 결과:

- favorites 테이블과 UI가 일치한다.
- 찜 수가 목록/상세에 반영된다.

## P0 관리자 테스트

### 11. 관리자 대시보드 접근

목적:

- admin role 사용자가 관리자 화면에 접근 가능한지 확인한다.

테스트:

- 관리자 계정 로그인
- `/admin` 접속
- 검수 대기, 게시 중, 신고, 사용자, 공식 링크 클릭 카드 표시 확인

기대 결과:

- 관리자 대시보드가 열린다.
- 공식 링크 클릭 수 카드가 표시된다.

### 12. 관리자 포스터 검수 목록

목적:

- `review`, `published`, `rejected` 상태 필터가 정상 동작하는지 확인한다.

테스트:

- `/admin/posters` 접속
- 검수 대기 필터 클릭
- review 포스터만 표시
- 게시 중 필터 클릭
- published 포스터만 표시
- 반려됨 필터 클릭
- rejected 포스터만 표시

기대 결과:

- 필터별 목록과 카운트가 일치한다.

### 13. 포스터 상세 미리보기 modal

목적:

- 승인/반려 전 상세 정보를 확인할 수 있는지 테스트한다.

테스트:

- `/admin/posters` 접속
- review 포스터에서 미리보기 클릭
- modal에 이미지, 제목, 기관, 요약, 카테고리, 링크 표시 확인
- modal에서 승인 클릭
- 포스터 상태가 published로 변경되는지 확인

기대 결과:

- modal 정보가 누락 없이 보인다.
- modal 내 액션이 정상 동작한다.

### 14. 단건 반려와 반려 사유

목적:

- 포스터를 반려할 때 사유가 저장되는지 확인한다.

테스트:

- review 포스터 선택
- 반려 버튼 클릭
- 반려 사유 입력
- 반려 처리
- rejected 필터에서 해당 포스터 확인
- 반려 사유 표시 확인

기대 결과:

- poster_status가 rejected로 변경된다.
- rejection_reason이 저장된다.
- admin_actions 로그가 생성된다.

### 15. 대량 승인

목적:

- 여러 review 포스터를 한 번에 승인할 수 있는지 확인한다.

테스트:

- `/admin/posters` review 필터 접속
- 전체 선택 또는 2건 이상 선택
- 선택 승인 클릭
- 선택 항목이 published로 변경되는지 확인
- admin_actions에 bulk metadata 기록 확인

기대 결과:

- 선택된 항목만 승인된다.
- 버튼 처리 중 중복 클릭이 방지된다.

### 16. 대량 반려

목적:

- 여러 review 포스터를 한 번에 반려할 수 있는지 확인한다.

테스트:

- review 포스터 2건 이상 선택
- 선택 반려 클릭
- bulk 반려 사유 입력
- 반려 처리
- rejected 필터에서 해당 항목 확인

기대 결과:

- 선택된 항목만 rejected가 된다.
- 반려 사유가 동일하게 저장된다.

### 17. 반려 포스터 완전 삭제

목적:

- rejected 포스터를 관리자 화면에서 완전 삭제할 수 있는지 확인한다.

테스트:

- rejected 필터 접속
- 반려 포스터 1건 선택
- 완전 삭제 클릭
- confirm 승인
- 목록에서 사라지는지 확인
- posters 테이블에서 삭제 확인
- admin_actions delete 로그 확인

기대 결과:

- rejected 상태의 포스터만 삭제 가능하다.
- published/review 포스터는 삭제 버튼 대상이 아니다.

## P1 크롤러/업로드 테스트

### 18. 크롤러 title exclusion

목적:

- 비포스터 제목이 수집 단계에서 제외되는지 확인한다.

fixture 예시:

- `RSS`
- `마포구 외국인관광 도시민박업 신청안내`
- `2026년 안심장비 지원사업 신청 안내`
- `마포장애인종합복지관 5월 휴관 안내`
- `📢 프로그램 게시요청 가이드 📢`

테스트:

- `getPostExclusionReason`에 제목 입력
- 제외 대상은 reason 반환 확인
- 포스터성 제목은 null 반환 확인

기대 결과:

- 명확한 비포스터 제목은 크롤러 결과에 들어가지 않는다.

### 19. 이미지 규칙 필터

목적:

- 포스터 이미지 후보만 남기는지 확인한다.

테스트:

- 작은 아이콘 이미지 후보 입력
- 로고 파일명 후보 입력
- 세로형 큰 이미지 후보 입력
- 가로 배너 후보 입력
- `selectBestPosterImage` 결과 확인

기대 결과:

- 아이콘/로고/버튼은 낮은 점수 또는 제외
- 세로형 포스터 후보가 우선 선택된다.

### 20. AI 포스터 이미지 분류 fallback

목적:

- OpenAI API key가 없거나 classifier가 꺼진 상황에서도 크롤러가 실패하지 않는지 확인한다.

테스트:

- `POSTER_IMAGE_CLASSIFIER=off`
- `OPENAI_API_KEY` 없음
- 크롤러 실행
- 분류기 disabled reason을 반환하고 진행되는지 확인

기대 결과:

- API 설정이 없어도 크롤러 전체가 죽지 않는다.

### 21. 크롤러 업로드 이미지 없음 제외

목적:

- 이미지 없는 포스터가 Supabase에 업로드되지 않는지 확인한다.

테스트:

- 이미지 없는 JSON fixture 업로드
- `이미지 없음 제외` 카운트 증가 확인
- posters insert가 발생하지 않는지 확인

기대 결과:

- thumbnail_url 없는 항목은 제외된다.

### 22. source_key 중복 처리

목적:

- 같은 게시글이 query parameter만 바뀌어도 중복으로 생성되지 않는지 확인한다.

fixture:

- `https://example.com/view?id=1&page=1`
- `https://example.com/view?id=1&page=2`
- `https://example.com/view?id=1&sortOrder=desc`

테스트:

- 첫 번째 업로드 성공
- 두 번째/세 번째 업로드는 중복 skip
- 기존 source_key가 정규화되어 업데이트되는지 확인

기대 결과:

- 같은 원본 게시글은 posters에 1건만 존재한다.

### 23. summary_short 생성

목적:

- 크롤러가 주요 내용을 깔끔하게 요약하는지 확인한다.

테스트:

- 신청기간, 대상, 문의가 포함된 본문 fixture 입력
- `summary_short`에 핵심 정보가 포함되는지 확인
- 너무 긴 문장과 중복 문구가 제거되는지 확인

기대 결과:

- 카드/상세에서 읽기 좋은 짧은 요약이 생성된다.

### 24. 카테고리 자동 분류

목적:

- 수집된 포스터가 적절한 카테고리와 연결되는지 확인한다.

테스트:

- 복지 키워드 포함 포스터 입력
- 채용 키워드 포함 포스터 입력
- 문화/교육 키워드 포함 포스터 입력
- `poster_categories`에 1~2개 카테고리 연결 확인
- 기존 중복 포스터에도 카테고리 보강 확인

기대 결과:

- category가 없는 크롤러 결과도 기본 분류가 생성된다.

## P1 사용자 여정 테스트

### 25. 회원가입 약관 UI

목적:

- 약관 동의 전 회원가입 버튼 비활성화 여부를 확인한다.

테스트:

- `/signup` 접속
- 이메일/비밀번호 입력 전 버튼 비활성 확인
- 필수 약관 동의 전 버튼 비활성 확인
- 필수 약관 동의 후 버튼 활성 조건 확인

기대 결과:

- 필수 약관을 동의해야 가입 진행 가능하다.

### 26. 온보딩 완료

목적:

- 신규 사용자가 온보딩을 완료해야 주요 페이지를 쓸 수 있는지 확인한다.

테스트:

- onboarding_completed false 사용자 로그인
- `/` 접속 시 `/onboarding` 이동
- 지역, 나이, 성별, 관심 카테고리 입력
- 완료 클릭
- `profiles.onboarding_completed` true 확인
- `/` 이동 확인

기대 결과:

- 온보딩 미완료 사용자는 온보딩으로 유도된다.
- 완료 후 정상 사용 가능하다.

### 27. 포스터 등록 요청

목적:

- 일반 사용자가 포스터 등록 요청을 제출할 수 있는지 확인한다.

테스트:

- 일반 사용자 로그인
- `/posters/request` 접속
- 설명 입력
- 위치 텍스트 입력
- 이미지 업로드 선택
- 제출
- `poster_requests.status = pending` 확인

기대 결과:

- 요청이 생성되고 관리자 검수 대상으로 들어간다.

### 28. 등록 요청 승인과 포인트 지급

목적:

- 관리자가 등록 요청을 승인하면 사용자에게 포인트가 지급되는지 확인한다.

테스트:

- 관리자 로그인
- `/admin/requests` 접속
- pending 요청 승인
- `poster_requests.status = approved`
- `point_logs`에 50포인트 로그 생성
- `profiles.points` 50 증가
- 사용자 마이페이지에서 포인트 표시 확인

기대 결과:

- 승인과 포인트 지급이 한 번에 처리된다.

### 29. 등록 요청 반려

목적:

- 관리자가 등록 요청을 반려할 수 있는지 확인한다.

테스트:

- `/admin/requests` 접속
- pending 요청 반려
- 반려 사유 입력
- `poster_requests.status = rejected`
- 포인트 미지급 확인

기대 결과:

- 반려된 요청은 포인트 지급이 없다.

## 메모

관리자/운영자/일반 사용자 권한을 포함한 테스트는 실제 Supabase 환경에 직접 쓰기를 하므로, 운영 DB가 아닌 테스트 프로젝트 또는 테스트 전용 계정/데이터로 분리하는 것이 안전하다.

크롤러 업로드 테스트는 실제 공공기관 사이트에 의존하지 않고 JSON fixture 기반으로 먼저 검증하는 것이 좋다. 실제 네트워크 크롤링은 별도 smoke test로 분리한다.
