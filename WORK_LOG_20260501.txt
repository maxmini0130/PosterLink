12. 2026-05-01 Mobile social login bridge fix
- Symptom:
  - Google/Kakao social login completed in native OAuth, but the WebView stayed logged out.
- Root cause:
  - apps/mobile/App.tsx wrote the session only to localStorage.
  - apps/web/lib/supabase.ts uses createBrowserClient(@supabase/ssr), which persists auth in base64url chunked cookies.
  - SESSION_BRIDGE_JS could not read those SSR cookies, so native and WebView auth state diverged.
- Fix:
  - apps/mobile/App.tsx
    - send the WebView to /auth/callback#access_token=...&refresh_token=... after native auth
    - parse base64url chunked auth cookies in SESSION_BRIDGE_JS
    - prefer cookie state over stale localStorage when syncing WebView auth back to native
  - apps/web/app/auth/callback/page.tsx
    - accept implicit-flow access_token/refresh_token from hash or query
    - call supabase.auth.setSession() in the browser callback page
    - scrub token params from the URL before redirecting
  - apps/web/app/login/page.tsx
    - add auth_callback_failed / set_session_failed error messages
- Verification:
  - pnpm --filter web lint OK
  - pnpm --filter web build OK
  - pnpm exec expo config --type public OK
  - Babel transform for apps/mobile/App.tsx OK
- Next:
  - commit and trigger a new EAS Android preview build

13. 2026-05-01 Mobile social login follow-up
- User report:
  - The new build still showed "소셜 로그인 세션 저장에 실패했습니다."
- Reassessment:
  - The failure was happening on the web callback page path, where the browser client tried to call supabase.auth.setSession().
  - That path is brittle for the app because native auth already has a valid session, and the WebView only needs the same session persisted in the format expected by @supabase/ssr.
- Change:
  - apps/mobile/App.tsx
    - stop routing native mobile auth through /auth/callback
    - write both localStorage and SSR-compatible auth cookies directly inside the WebView
    - use base64url encoding and 3180-byte chunking compatible with @supabase/ssr cookie storage
    - clear stale auth cookie chunks before writing the new session
  - apps/web/app/login/page.tsx
    - rewrite the page in clean UTF-8 Korean
    - include ?msg=... details in toast errors for faster diagnosis if another auth error appears
- Verification:
  - pnpm --filter web lint OK
  - Babel transform for apps/mobile/App.tsx OK
- Next:
  - commit, push, and issue a new Android preview build

14. 2026-05-01 new_match push hardening
- Reason:
  - Admin publish flow called notify-new-match with an anon fallback token when no user session was available.
  - The Edge Function itself did not verify the caller role.
  - Re-triggering the function could resend the same push notifications for the same poster.
- Changes:
  - apps/web/app/admin/posters/page.tsx
    - removed anon-key fallback from notify-new-match calls
    - require a real user session token for the Edge Function call
    - show a clearer post-approval push summary in the admin UI
  - supabase/functions/notify-new-match/index.ts
    - verify Authorization header
    - resolve the caller with auth.getUser()
    - allow only admin / super_admin roles to trigger push delivery
    - send pushes only for notifications where push_sent_at is null
    - mark successfully pushed notifications with push_sent_at
  - supabase/migrations/20260501010000_harden_notify_new_match_push.sql
    - add notifications.push_sent_at
    - add pending-push index for new_match notifications
